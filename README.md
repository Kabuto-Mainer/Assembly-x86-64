# Собственная реализация функции printf (x86-64, NASM)

# 1. Цель работы

Написание собственного аналога стандартной функции `printf` с реализацией следующих спецификаторов вывода: `%d`, `%g`, `%s`, `%c`, `%b`, `%o`, `%x` на языке `Assembler` под архитектуру `x86-64`.

# 2. Архитектура

Функция реализована на `NASM Assembler` в виде отдельного файла под ядро операционной системы `Linux`.

Выходной буфер расположен на стеке и расширяется вниз, по мере увеличения длины. Верхним ограничением длины является верхушка стека.
При увеличении объема данных верхушка стека смещается вниз.

Обработка форматной строки происходит следующим образом:
- Считывается символ из форматной строки.
- Если это `%`, то происходит:
    - Считывание следующего символа.
    - Сравнение его с символами поддерживаемых спецификаторов.
    - Если это также `%`, то в буфер записывается `%`.
    - Если это символ спецификатора, то аргумент соответствующего типа извлекается из:
        - целочисленных регистров,
        - векторных регистров,
        - стека,
    в зависимости от количества уже обработанных аргументов каждого типа.
    После этого значение выводится в соответствующем формате.

    - Если это не `%` и не спецификатор, то это считается ошибкой и происходит преждевременный вывод содержимого. Дальнейшая форматная строка не обрабатывается, а функция просто завершает свою работу.

Используется системный вызов write (syscall 1):
- RAX = 1
- RDI = 1 (stdout)
- RSI = указатель на буфер
- RDX = длина буфера

Данные выводятся в стандартный файл вывода.

Аргументы передаются в соответствии с `System V AMD64 ABI`:
- Первые 6 целочисленных аргумента: `RDI` (1), `RSI` (2), `RDX` (3), `RCX` (4), `R8` (5), `R9` (6).
- Первые 8 чисел с плавающей точкой: `XMM0` (1), `XMM1` (2), ..., `XMM7` (8).
- Аргументы, не поместившиеся в регистры, передаются через стек. Стек перед вызовом выравнивается по границе 16 байт.

Возвращаемое значение — количество обработанных аргументов (нестандартное поведение).

# 3. Ограничения

Ограничение реализации: значение регистра `AL`, которое по System V AMD64 ABI используется для variadic-функций и содержит число задействованных регистров `XMM`, не анализируется. Реализация предполагает корректное расположение аргументов в `XMM`-регистрах и может работать некорректно в нетипичных сценариях вызова.


# 4. Использование

Для использования соответствующей функции необходимо объявить:
`int my_printf(char *format, ...)`.
При линковке своей программы нужно добавить `my_printf.o`.

Функция поддерживает следующие спецификаторы:
- `%d` - вывод 4-байтного целого числа со знаком.
- `%c` - вывод 1-байтового символа в формате `ASCII`.
- `%s` - вывод строки из 1-байтовых символов в формате `ASCII`. Концом строки считается нулевой символ '\0'.
- `%g` - (нестандартное поведение!) вывод числа с плавающей точкой с двойной точностью (аналог `%f`).
- `%b` - вывод 1-байтового значения в двоичной системе счисления. Длина выводимого значения всегда `8` символов.
- `%o` - вывод 4-байтового беззнакового числа в восьмеричной системе счисления.
- `%x` - вывод 4-байтового беззнакового числа в шестнадцатеричной системе счисления.

Для работы с функцией используется `Makefile` со следующими командами:
- `make build I=my_printf` - создание объектного файла.
- `make link I=my_printf` - линковка объектного файла и выдача прав запуска для получившегося бинарного файла.

Для подключения функции в свою программу необходимо при линковке файлов указать объектный файл функции.

Пример:
`gcc test.c my_printf.o -o test`.

# 5. Примеры вызова

`my_printf("%s %g %s", "Poltorashka eats", 3.14, "mouses")` -> `Poltorashka eats 3.140000 mouses`

`my_printf("%x %b %o", 0xeda, 7, 10)` -> `eda 00000111  12`
